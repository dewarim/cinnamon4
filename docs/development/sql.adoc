Developer notes on non-obvious SQL queries in the codebase.

== getFolderByIdWithAncestors (FolderMapper.xml)

Returns a target folder and all of its ancestors, ordered from root to target.
Used by `FolderDao.getFolderByIdWithAncestors` and ultimately by `FolderService.resolvePath`,
which calls `ancestors.getLast()` to get the target folder itself.

=== The CTE: WITH RECURSIVE fp

The CTE builds a path table for the *entire* folder tree in two steps.

*Base case (anchor):*

[source,sql]
----
select id, parent_id, name, array[id], 0
from folders f
where f.parent_id is null
----

Starts at the one root folder (no parent). Initialises:

* `ids = ARRAY[root_id]` — just its own ID
* `depth = 0`

*Recursive case:*

[source,sql]
----
select child.id, child.parent_id, child.name,
       array_append(ids, child.id), fp.depth + 1
from folders child join fp on child.parent_id = fp.id
where child.parent_id is not null
----

For every child of the current frontier, appends the child's own ID to the parent's `ids`
array and increments depth. The recursion expands until no more children exist.

After the CTE runs, every folder row in `fp` carries its *full ancestry path as an array*,
from root down to itself:

[cols="1,2,1"]
|===
| folder    | ids           | depth

| root      | `[1]`         | 0
| home      | `[1, 2]`      | 1
| admin     | `[1, 2, 4]`   | 2
| documents | `[1, 2, 4, 7]`| 3
|===

=== The WHERE clause: filtering to ancestors of #{id}

[source,sql]
----
WHERE array_length(fp.ids, 1) <= (select array_length(ids, 1) from fp where id = #{id})
  AND fp.ids <@ (select ids from fp where id = #{id})
----

Say `#{id}` is the `admin` folder, whose `ids = [1, 2, 4]` (length 3).

* *Condition 1* — `array_length(fp.ids, 1) \<= 3`: eliminates all folders deeper than admin
  (e.g. `documents` with length 4).
* *Condition 2* — `fp.ids <@ [1, 2, 4]`: the `<@` operator means _"is contained by"_.
  `fp.ids` must be a subset of admin's path array.

Only three arrays satisfy both conditions:

[cols="2,2,1,2"]
|===
| folder          | ids         | length ≤ 3? | subset of `[1,2,4]`?

| root            | `[1]`       | ✓           | ✓
| home            | `[1, 2]`    | ✓           | ✓
| admin           | `[1, 2, 4]` | ✓           | ✓
| sibling folder  | `[1, 2, 5]` | ✓           | ✗ (5 ∉ admin's path)
|===

This selects exactly the target folder plus all its ancestors — nothing more.

=== ORDER BY fp.depth ASC

Without this, PostgreSQL can return rows in any order.
The caller uses `ancestors.getLast()` to get the target folder (the deepest entry),
so ordering root→target is essential.

NOTE: The absence of this `ORDER BY` was a bug that caused non-deterministic path
resolution: PostgreSQL would sometimes return the home folder last instead of the
intended target, making `resolvePath` navigate to the wrong folder.

=== The JOIN back to folders

[source,sql]
----
FROM folders fo JOIN fp ON fp.id = fo.id
----

The CTE only stores `id`, `parent_id`, `name`, `ids`, `depth`.
The JOIN pulls in the full row (`acl_id`, `owner_id`, `metadata_changed`, etc.)
for each matched folder.
